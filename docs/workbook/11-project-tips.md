# [11] 설계의 묘수: 프로젝트 유지 관리 및 확장 팁

하나의 시스템을 설계하고 끝이 아닙니다. 이 프로젝트 베이스(BCC)를 기반으로 변형(Variation)을 만들고자 할 때 추천하는 운영 방침입니다.

## 1. 지표(Metrics)를 추적하라
Sci-Fi HUD 화면 상단, 혹은 브라우저 디버그 콘솔에 노출되는 `FPS` 와 `Latency(ms)` 지표는 단순한 멋이 아닙니다.
- **성능 저하의 시그널**: 만약 모델 인식 모듈 로직을 수정했는데, FPS가 30에서 5로 뚝 떨어졌다면 코드를 무조건 즉시 롤백하십시오 (백그라운드 스레드에 메인 스레드가 블럭당한 상태입니다.)

## 2. 파일 I/O 동시성의 함정 조심 (Database Locks)
현재 BCC는 데이터 엔진으로 `SQLite3`를 사용합니다. 일반적인 JSON 보다는 압도적으로 훌륭한 동시성을 보장하지만, 수십 명의 사용자가 동시에 POST API로 얼굴을 등록하려 들면 DB Lock (`Database is locked` 에러)이 발생할 수 있습니다.

**[확장 제안]** 만약 엔터프라이즈 환경 다중 접속 클라이언트용으로 바꾸려면 `SQLite` 를 MySQL 또는 PostgreSQL 등의 완전한 RDBMS 모델로 교체하거나 컨넥션 풀링(Connection Pooling) 로직을 추가해야 합니다.

## 3. 리소스의 한계치 통제 (CPU / 발열 억제력)
이 프로그램이 돌아가면 온 디바이스 모델 연산인 만큼 노트북 팬(Fan)이 비명을 지르기 시작할 겁니다.
- **방어 로직의 중요성**: `/api/stream` 이 요청받지 않는 유휴 단계라면, AI 모델 인퍼런스가 프레임 낭비를 일으키지 않도록 로직을 일시 멈추도록(Hibernate) 고도화하는 것을 추천합니다. 

이러한 설계의 디테일과 한계 비용을 염두에 두면 코더(Coder)에서 시스템 엔지니어(Systems Engineer)로 도약할 수 있습니다.
